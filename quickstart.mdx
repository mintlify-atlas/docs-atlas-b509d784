This guide walks you through creating a controllable moving cube from scratch. By the end you will have a working entity in the editor that you can move with the arrow keys or WASD in play mode.

You will define three variants ‚Äî `Position`, `Speed`, and `Cube` ‚Äî wire them together using the Query API, create an entity in the editor, and run it in play mode.

## Project structure

Game code belongs in specific directories so the preparser and build system can find it:

```plaintext
zeytin/
‚îú‚îÄ‚îÄ editor/
‚îÇ   ‚îú‚îÄ‚îÄ include/
‚îÇ   ‚îú‚îÄ‚îÄ source/
‚îÇ   ‚îî‚îÄ‚îÄ build.sh
‚îú‚îÄ‚îÄ engine/
‚îÇ   ‚îú‚îÄ‚îÄ include/
‚îÇ   ‚îÇ   ‚îî‚îÄ‚îÄ game/           ‚Üê Game header files go here
‚îÇ   ‚îú‚îÄ‚îÄ source/
‚îÇ   ‚îÇ   ‚îî‚îÄ‚îÄ game/           ‚Üê Game implementation files go here
‚îÇ   ‚îî‚îÄ‚îÄ build.sh
‚îî‚îÄ‚îÄ shared_resources/
    ‚îú‚îÄ‚îÄ entities/
    ‚îî‚îÄ‚îÄ variants/
```

Headers go under `engine/include/game/` and implementations go under `engine/source/game/`. The preparser scans these directories to generate variant registration code and detect Query-based dependencies.

## Steps

<Steps>
  <Step title="Create the Position variant">
    Create `engine/include/game/position.h`. This variant stores the 2D world position of any entity.

    ```cpp
    #pragma once

    #include "variant/variant_base.h"

    class Position : public VariantBase {
        VARIANT(Position)  // Required macro for engine-side registration

    public:
        Position(float x, float y) : x(x), y(y) {}

        float x = 0; PROPERTY()  // Marks this field as an editable property
        float y = 0; PROPERTY()
    };
    ```

    <Note>
      `VARIANT(TypeName)` generates the boilerplate the engine needs to register and reflect the class via RTTR. `PROPERTY()` marks a field as visible and editable in the editor ‚Äî both at design time and during play mode. These are empty macros at the language level; their significance is to the preparser and the editor.
    </Note>
  </Step>

  <Step title="Create the Speed variant">
    Create `engine/include/game/speed.h`. This variant holds a single scalar speed value.

    ```cpp
    #pragma once

    #include "variant/variant_base.h"

    class Speed : public VariantBase {
        VARIANT(Speed);

    public:
        float value; PROPERTY()
    };
    ```
  </Step>

  <Step title="Create the Cube variant header">
    Create `engine/include/game/cube.h`. The `Cube` variant owns the rendering and input logic.

    ```cpp
    #pragma once

    #include "variant/variant_base.h"
    #include "game/position.h"

    class Cube : public VariantBase {
        VARIANT(Cube);

    public:
        float width = 50.0f; PROPERTY()
        float height = 50.0f; PROPERTY()
        Color color = RED; PROPERTY()  // Color is a Raylib struct ‚Äî nesting is supported

        void on_init() override;         // Called once after construction and property deserialization
        void on_update() override;       // Called every frame (editor and play mode)
        void on_play_update() override;  // Called every frame while in play mode

    private:
        void handle_input();
    };
    ```

    The lifecycle methods available on every variant are:

    | Method | When it runs |
    |---|---|
    | `on_init()` | Once, after construction and property deserialization |
    | `on_play_start()` | Once, when entering play mode |
    | `on_early_update()` | Every frame, before `on_update()` |
    | `on_update()` | Every frame (editor and play mode) |
    | `on_play_update()` | Every frame, only in play mode |
    | `on_play_late_update()` | Every frame in play mode, after `on_play_update()` |
  </Step>

  <Step title="Implement the Cube logic">
    Create `engine/source/game/cube.cpp`. This is where `Position` and `Speed` are queried and used.

    ```cpp
    #include "game/cube.h"
    #include "game/speed.h"
    #include "core/query.h"
    #include "core/raylib_wrapper.h"

    #include "remote_logger/remote_logger.h"  // Send log messages to the editor console

    void Cube::on_init() {
        log_info("Cube is initialized");
    }

    void Cube::on_update() {
        // Get the position component
        auto& position = Query::get<Position>(this);

        // Draw the cube
        draw_rectangle(
            position.x - width / 2,
            position.y - height / 2,
            width,
            height,
            color);
    }

    void Cube::on_play_update() {
        // Handle user input during play mode
        handle_input();
    }

    void Cube::handle_input() {
        auto& position = Query::get<Position>(this);
        // Using Query::read for Speed as it's read-only
        const auto& speed = Query::read<Speed>(this);

        float delta_time = get_frame_time();
        float movement_speed = speed.value;

        if (is_key_down(KEY_LEFT) || is_key_down(KEY_A)) {
            position.x -= movement_speed * delta_time;
        }
        if (is_key_down(KEY_RIGHT) || is_key_down(KEY_D)) {
            position.x += movement_speed * delta_time;
        }
        if (is_key_down(KEY_UP) || is_key_down(KEY_W)) {
            position.y -= movement_speed * delta_time;
        }
        if (is_key_down(KEY_DOWN) || is_key_down(KEY_S)) {
            position.y += movement_speed * delta_time;
        }
    }
    ```

    - `Query::get<T>(this)` returns a mutable reference to a variant on the same entity as `this`.
    - `Query::read<T>(this)` returns a `const` reference ‚Äî use it when you only need to read data.

    <Note>
      The editor's preparser scans your source files and detects `Query::get` and `Query::read` calls inside `Cube`. It treats `Position` and `Speed` as dependencies of `Cube`, so when you add the `Cube` variant to an entity in the editor, `Position` and `Speed` are **automatically added as well**.

      You can disable this auto-detection per variant with `IGNORE_QUERIES()`, or declare dependencies explicitly with `REQUIRES(Position, Speed)` ‚Äî both are optional and work independently of each other.
    </Note>
  </Step>

  <Step title="Create an entity in the editor">
    Launch the editor from the `editor/` directory:

    ```bash
    ./bin/Debug/ZeytinEditor
    ```

    In the **Hierarchy** panel on the left, click **+ Create New Entity** and enter a name ‚Äî for example, `TestEntity`.

    This creates a design-time entity. Design-time entities are created in the editor and constructed by the engine at startup when it loads the scene, as opposed to runtime entities that are instantiated while the game is running.
  </Step>

  <Step title="Add the Cube variant">
    Right-click **TestEntity** in the hierarchy and select **Add Variant**, then choose **Cube**.

    Because the editor detected `Query::get<Position>` and `Query::read<Speed>` in `cube.cpp`, it adds `Position` and `Speed` to the entity automatically alongside `Cube`.

    Set initial property values in the inspector:

    - `Position.x` and `Position.y` ‚Äî place the cube somewhere visible (e.g., `400`, `300`)
    - `Speed.value` ‚Äî set a non-zero value (e.g., `200`)
    - `Cube.width` and `Cube.height` ‚Äî adjust the size if you like (e.g., `80`, `80`)
    - `Cube.color` ‚Äî leave as `RED` or pick another Raylib color
  </Step>

  <Step title="Start the engine and enter play mode">
    Click the **Start Engine** button at the top of the editor window.

    This triggers a rebuild of the engine binary using the `EDITOR_MODE` configuration and then launches it. The engine view in the centre panel becomes active. Check the **Console** panel at the bottom for build errors or log output ‚Äî you should see `Cube is initialized` from `on_init`.

    Click **Play** in the editor toolbar to enter play mode.

    Use the arrow keys or **W A S D** to move the cube. The `on_play_update` method runs every frame and updates `Position` via the input handler.

    <Tip>
      You can edit property values while in play mode ‚Äî changes take effect immediately in the running engine. However, they are **not saved** back to the scene. Exit play mode to return to the original design-time values and make permanent edits.
    </Tip>
  </Step>
</Steps>
