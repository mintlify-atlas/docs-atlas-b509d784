title: "Preparser"
description: "How the Clang-based preparser generates reflection code for your variants"
---

The preparser is a standalone Clang AST tool that reads your variant source files and generates two kinds of output: RTTR registration code that the engine compiles, and `.component` descriptor files that the editor reads to know which variants exist and what properties they expose.

You do not need to run the preparser manually under normal use — it runs automatically whenever you click **Start Engine** in the editor.

## Why it exists

Zeytin uses [RTTR](https://www.rttr.org/) (Run-Time Type Reflection) to serialise variant instances and expose them to the editor's inspector panel. Every variant type and every `PROPERTY()`-marked field must be registered with RTTR before the engine starts.

Writing that registration code by hand is repetitive and error-prone. The preparser automates it: you annotate your fields with `PROPERTY()`, and the preparser generates the boilerplate for you.

## How it works

<Steps>
  <Step title="Load the compilation database">
    The preparser takes a single argument: the path to the directory containing `compile_commands.json`. It loads this file using Clang's `CompilationDatabase` API to understand the include paths and compiler flags used for your project.

    ```bash
    ./preparser <compile_commands_dir>
    ```

    `compile_commands.json` is generated by Premake when you run `premake5 gmake`. If this file is missing, the preparser will exit with an error before scanning any source files.
  </Step>

  <Step title="Scan game source files">
    The preparser locates the `engine/source/game/` directory relative to the compilation database path, then recursively collects every `.cppm` file it finds there. These are your variant module files.
  </Step>

  <Step title="Find classes derived from Component">
    A Clang `ASTMatchFinder` runs over the collected files, searching for every `CXXRecordDecl` that is derived from the `Component` base class. Each matching class becomes a `ComponentInfo` entry.

    For each class, the preparser:
    - Records the class name and source file path
    - Walks all fields and identifies those that carry a `PROPERTY()` annotation
    - Checks for `IGNORE_QUERIES()` and `REQUIRES()` macro hints that affect dependency resolution
  </Step>

  <Step title="Generate RTTR registration code">
    For each variant, the preparser writes a `.cpp` file under `engine/source/game/generated/rttr_register/`. The generated file imports the variant's module and calls the RTTR registration API:

    ```cpp
    #include "rttr/registration.h"

    import game.cube;

    RTTR_REGISTRATION
    {
        rttr::registration::class_<Cube>("Cube")
            (rttr::metadata("is_component", true))
            .constructor<>()(rttr::policy::ctor::as_object)
            .property("width", &Cube::width)
            .property("height", &Cube::height)
            .property("color", &Cube::color)
            .property("entity_id", &Cube::entity_id)
                (rttr::metadata("is_hidden", true));
    }
    ```

    The `entity_id` property is always added with `is_hidden` metadata so the editor does not display it in the inspector.
  </Step>

  <Step title="Export .component descriptor files">
    The preparser also writes a JSON-formatted `.component` file for each variant to `shared_resources/components/`. The editor reads these files at startup to populate the variant picker and the inspector panel.

    A `.component` file includes:
    - The variant's type name
    - The name and default value of every `PROPERTY()` field
    - Dependency metadata derived from `REQUIRES()` and `IGNORE_QUERIES()` hints
  </Step>

  <Step title="Remove orphaned generated files">
    After generating all output, the preparser compares the set of files it just wrote against everything already present in `engine/source/game/generated/rttr_register/` and `shared_resources/components/`. Any `.cpp` or `.component` file that no longer corresponds to a known variant is deleted automatically.
  </Step>
</Steps>

## The PROPERTY() macro

`PROPERTY()` is defined as an empty macro at compile time — it produces no code. Its purpose is to act as a marker that the preparser recognises from the AST to identify which fields to include in the generated RTTR registration.

```cpp
float width = 50.0f; PROPERTY()
Color color = RED;   PROPERTY()  // Raylib structs are also supported
```

## The VARIANT() macro

`VARIANT(TypeName)` expands to registration boilerplate at compile time that the engine needs to construct and manage variants. The preparser also reads this macro from the AST when identifying component classes, using it as a secondary signal alongside the `Component` base class derivation.

## IGNORE_QUERIES() and REQUIRES()

Both macros expand to nothing at compile time. The preparser parses them from the AST and encodes the resulting dependency metadata into the `.component` descriptor files. The editor reads that metadata to decide which variants to auto-add when you attach a component.

- `IGNORE_QUERIES()` — tells the editor not to analyse `Query::get` and `Query::read` calls for this variant.
- `REQUIRES(A, B, ...)` — declares an explicit list of variants that must be present on the same entity.

## Running the preparser manually

If you need to run the preparser outside of the normal editor flow:

```bash
./preparser path/to/engine/build
```

The argument must be the directory that contains `compile_commands.json`, which is the `engine/build/` directory after running `premake5 gmake`.

<Warning>
  The preparser requires a valid `compile_commands.json`. Always run `premake5 gmake` inside `engine/` before the first build or after changing `premake5.lua`. Without this file the preparser exits immediately and no code is generated.
</Warning>

## Common questions

<AccordionGroup>
  <Accordion title="What if I rename or delete a variant?">
    The preparser detects this automatically during the orphan-cleanup step. When a source `.cppm` file is deleted or a class name changes, the previously generated `.cpp` and `.component` files no longer appear in the expected-files set. The preparser removes them so the engine does not try to compile stale registration code.

    If the editor is open when you delete a variant, restart it after the next engine build so it re-reads the updated `shared_resources/components/` directory.
  </Accordion>
  <Accordion title="Why does the preparser only scan .cppm files?">
    Variant classes in Zeytin are defined in C++ module interface units (`.cppm`). The preparser filters for this extension specifically because module interface files are self-contained and carry the `export module` declaration the preparser uses to derive the module name for the generated `import` statement.
  </Accordion>
  <Accordion title="Can I add a property of a custom struct type?">
    Raylib struct types such as `Color` are supported as `PROPERTY()` fields — the quickstart example uses `Color color = RED; PROPERTY()` and the editor displays it correctly. Support for other custom struct types depends on whether RTTR can reflect them; if a struct is not registered with RTTR, the editor will not be able to display or edit it.
  </Accordion>
</AccordionGroup>
